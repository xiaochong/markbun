# Ideation: MarkBun v0.6.0 AI Support

## Codebase Context

- **Project**: MarkBun — Typora-like desktop markdown editor (Electrobun: Bun main process + WebView renderer)
- **Stack**: React + TypeScript + Tailwind + Milkdown/ProseMirror editor
- **Current version**: v0.5.0
- **Key architecture**: RPC pattern (types.ts -> index.ts -> electrobun.ts), Zod settings schema, sidebar tabs, ProseMirror decoration system
- **Existing features**: Backup system (3-layer), Export (HTML/PNG/PDF), i18n (8 locales), file explorer, outline view, quick-open, settings dialog with 5 tabs
- **pi-mono SDK**: Unified multi-provider LLM API. Streaming events (text_delta, toolcall_start, done, error), tool calling with TypeBox schemas, 22+ providers (including Ollama), JSON-serializable Context objects, cost tracking per request. Works in Bun and browser.

## Ranked Ideas

### 1. AI Settings & Provider Configuration
**Description:** New "AI" tab in SettingsDialog. Manage API keys stored in `~/.config/markbun/ai-keys.json` (not in settings.json). Provider/model selection with connectivity test button. Support Ollama local-only mode via checkbox ("Only use local models"). Use pi-ai's `getProviders()`/`getModels()` to dynamically list available models based on configured keys. Keep initial schema minimal: provider name, model ID, API key path, base URL override.
**Rationale:** Foundation for all AI features. The Zod schema + SettingsDialog tab + `mergeWithDefaults` pattern is already well-established. Ollama support enables zero-config local AI.
**Downsides:** API key storage security needs care. Initial schema should stay minimal — do not enumerate all 22+ providers in UI.
**Confidence:** 98%
**Complexity:** Low
**Status:** Unexplored

### 2. Streaming RPC Bridge
**Description:** Build a message-based (not request/response) channel from Bun main process to WebView for streaming AI events. Map pi-ai events (`text_delta`, `toolcall_start`, `done`, `error`) to typed RPC messages. Main process holds API keys and runs LLM calls; renderer only receives rendering events. May need 50ms buffering for high-frequency message bursts.
**Rationale:** Hard prerequisite. Existing Electrobun RPC is request/response with 30s timeout — cannot stream tokens over it. The `messages` channel in `MarkBunRPC.bun.messages` already supports one-way push from main to renderer.
**Downsides:** Need to verify Electrobun message throughput performance. Four event types sufficient for v0.6.0.
**Confidence:** 95%
**Complexity:** Medium
**Status:** Unexplored

### 3. AI Chat Panel (Right Sidebar)
**Description:** Independent right-side panel, separate from the left sidebar (files/outline/search). Users can see file navigation AND chat simultaneously. AI accesses the document via pi-ai tool calls: `read_document`, `read_selection`, `read_outline`, `insert_at_cursor`, `replace_selection`. System prompt auto-includes document context (full text for short docs, outline + current section for long docs). Session management: reset conversation, select historical conversations per file, create new sessions. Conversations stored in `~/.config/markbun/ai-sessions/<fileHash>/` with multiple sessions per file.

**Layout:**
```
┌──────────┬────────────────────────┬──────────┐
│  Left    │                        │  Right   │
│  Sidebar │   Editor (ProseMirror) │  AI Chat │
│ ────────  │                        │ ────────  │
│ Files    │                        │ [Session▼]│
│ Outline  │                        │ ────────  │
│ Search   │                        │ History   │
│          │                        │ ...       │
│          │                        │ ────────  │
│          │                        │ [Input]   │
└──────────┴────────────────────────┴──────────┘
```

**Rationale:** The core user-facing feature of v0.6.0. Most general — users can rewrite, translate, explain, summarize, generate anything via natural language. Right-side placement keeps navigation and AI visible simultaneously. Tool bridge lets AI act on the editor directly (not just suggest text). Session management enables persistent, resumable AI collaboration.
**Downsides:** Need to design chat UI (use pi-web-ui components or custom build). Tool call round-trip latency needs optimization. Multi-session per file adds storage complexity.
**Confidence:** 92%
**Complexity:** High
**Status:** Unexplored

### 4. Selection-Aware AI Actions
**Description:** Select text → right-click context menu shows AI actions: Improve Writing, Make Concise, Translate, Explain. Action results stream into the AI Chat Panel (#3) with a "Replace Selection" button. Clicking applies the change via ProseMirror transaction (supports Undo). Before any AI modification, auto-call `createVersionBackup()` from existing backup system as safety checkpoint.
**Rationale:** Highest-frequency AI use case in a writing tool. `getSelectedMarkdown()` and context menu infrastructure already exist. Streaming to Chat Panel (not inline replacement) avoids complex ProseMirror diff decoration engineering.
**Downsides:** Depends on Chat Panel (#3). Two features are tightly coupled.
**Confidence:** 90%
**Complexity:** Medium
**Status:** Unexplored

### 5. AI Session Persistence
**Description:** Leverage pi-ai's JSON-serializable `Context` objects. Store conversation history per file in `~/.config/markbun/ai-sessions/<hash>/session_<id>.json`. Auto-restore last active session when file opens. Handle schema changes gracefully (start fresh if deserialization fails).
**Rationale:** pi-ai Context is naturally serializable — near-zero cost. Transforms AI from stateless tool to collaborative partner that remembers prior discussions. Storage pattern matches existing `backup.ts` (`getPathHash()`).
**Downsides:** Need to handle Context schema forward-compat across pi-ai versions.
**Confidence:** 88%
**Complexity:** Low-Medium
**Status:** Unexplored

### 6. i18n-Aware AI Prompts
**Description:** Auto-inject document language info into system prompts (based on i18n setting or content detection). Prevents "improve this Chinese text" producing English output — a common and catastrophic failure mode. Implementation is minimal (~5 lines in system prompt construction).
**Rationale:** MarkBun's core user base includes Chinese writers. Without language awareness, AI features would produce unacceptable results for this segment.
**Downsides:** None meaningful.
**Confidence:** 95%
**Complexity:** Low (implementation detail, not standalone feature)
**Status:** Unexplored

## Dependency Graph

```
[1: AI Settings] → [2: Streaming Bridge] → [3: AI Chat Panel (Right)]
                                                    ↓
                         [4: Selection Actions] ← [5: Session Persistence]
                                                    ↓
                                             [6: i18n Prompts]
```

Minimum viable AI = **1 + 2 + 3** (settings → streaming → chat panel), then layer 4, 5, 6.
